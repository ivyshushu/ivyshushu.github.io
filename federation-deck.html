Current solution: Free monad interpreter

Opportunity
  • The data registry we own today is the seed of a federated data mesh platform.

Gaps
  • No access governance: Any product team can consume a domain's data field without the owner's knowledge.
  • No data versioning: Product data derived from domain data has no versioned contract.
  • Resilience risk: Single point of failure across all products
  • Operational overhead: On-call burden concentrated on platform team
Slide 2 — Strategic Pivot
3-layer data governance architecture
Combining Uber DOMA's domain ownership with Airbnb Viaduct's query optimization — with no centralized service in the critical path.
Uber DOMA
Domain ownership & gateways
Organizes 2,200 services into 70 domains. Gateways shield consumers from churn. Extensions allow custom logic injection.
Airbnb Viaduct
Centralized query engine
Single GraphQL schema with team-owned modules, re-entrant resolvers, serverless transformations, and field-level observability.
Layer 1
Domain layer (DOMA-inspired)
Domain teams own raw data and expose typed schemas. Define join keys, extension points, SLA timeouts, and on-call handlers. Internal implementation hidden behind the schema contract.
Layer 2
Product layer (GQL Extensions + Data Ownership)
Product teams extend domain schemas with product-specific derivations. Extensions are declarative, approved by domain teams, and transformation logic is sharable.
Layer 3
Presentation layer (Viaduct-inspired SDK)
Platform SDK generates a resolved data graph at build time. Resolver data query, batching, fan-out and caching run as a library — no centralized service to operate.
Benefits
  • Clear ownership: domain / product / presentation
  • Fine-grained access control
  • Dev experience: presentation service queries data via SDK
  • Failure blast radius reduced to single product
Tradeoffs & mitigations
  • SDK becomes a critical shared dep → strong versioning strategy
  • Annotation ceiling in Thrift → GQL schema with custom directives
  • Schema governance overhead → invest in tooling early
  • Decentralized fan-out → SDK-level instrumentation + domain SLAs

Appendix — Schema Design
3-layer schema in practice
Thrift stays the RPC wire contract. GraphQL — generated from Thrift — is the canonical governance layer stored in the schema registry, with directives encoding ownership, versioning, and extension rules.
Layer 1 — Domain codegen from Credit Report service IDL · owned by credit-eco-team
type CreditReport @domain(owner: "credit-eco-team", sla: {p95LatencyMs: 80}, extensions: ["core-product-team"]) {
  creditScore: Int  @field(joinKey: "userId", version: "1.0.0")
}
version is bumped by domain team whenever implementation changes — the schema needs to be backward compatible. CI enforces the bump on every code change to fields backing this type monitored by AI.
↓ product team registers extension, approved by domain team
Layer 2 — Product Core Product team · extends CreditReport.creditScore
extend type CreditReport {
  creditProfile: CreditProfile @extension(
    dependsOn: ["CreditReport.creditScore@1.0.0"]
  )
}

type CreditProfile {
  equifaxCreditScore: Int @derived(source: "CreditReport.creditScore", fact: "creditScore", recipe: "@ck/fact-cookbook@1.0.0/score-converter")
}
dependsOn pins domain field versions. If CreditReport.creditScore bumps to 1.0.1, the registry flags this extension as stale and notifies the core product team before it reaches production.
↓ SDK resolves fetch plan at runtime
Layer 3 — Presentation query declared in Dashboard service and mapped to facts in registry
query Dashboard($userId: ID!) {
  creditReport(userId: $userId) {
    creditProfile {
      equifaxCreditScore
    }
  }
}
