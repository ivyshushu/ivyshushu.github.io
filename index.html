Why This Became a Liability

CENTRALIZED ARCHITECTURE
Architectural Fragility
Any outage cascades to ALL content targeting across every product surface
Maintainability Barrier
Free monad program composed with cats and Finagle futures, interpreted by a lazy batch runner. Requires end-to-end understanding of the evaluation pipeline to maintain safely -- most engineers on the team could not confidently modify this code.
Security Blast Radius
One service with read-access to ALL domain data -- credit, account, behavior, ML features
Cross-team Coupling
Every data point definition needs to onboard to the central registry, adding friction for all product teams.
Rigid Data Types
Fixed types: Number, String, Boolean, Date. No complex structures. Adding a type = ADT + evaluator + serialization changes.

06

Future: Agentic Data Layer

WHAT THIS UNLOCKS
Traditional Approach: Tool-per-Service RAG
Each domain service requires a hand-built tool definition. The LLM picks which tool to call, but cannot compose across services in a single query. Adding a new data source means writing a new tool, prompt-engineering the selection logic, and testing tool interactions. Scales linearly with the number of services.
Agentic Data Layer: Schema-Driven Queries
Domain services and data points are described as a GraphQL schema. The LLM reads the schema and generates a real GQL query expressing exactly what it needs -- across any combination of services, in one request. The SDK resolves each field via its registered fetcher, with caching and batching built in. Adding a data source = extending the schema. No new tools, no prompt changes.
The Federated Data SDK already maps typed fields to registered fetchers with request-scoped caching. GraphQL is the natural interface: the schema becomes the LLM's "tool documentation," and the query becomes its structured output -- composable, type-safe, and executable without building bespoke tools for every data source.
