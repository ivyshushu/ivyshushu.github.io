Key Challenges

1. Cross-Runtime Ownership is Novel
The framework spans three runtimes with interleaved Rust/JNI/NAPI layers and data crosses the boundary with non-obvious ownership transfer.
2. Complex Codebase, No Guide
The primary owner was unavailable during the investigation, so we had to learn the codebase with little help.
3. No Native Profiling Experience
We had to learn native profiling from scratch to get visibility into the right layer.
02
Local-First Debug Approach

Reproduce, Profile, Fix, Prove

PHASE 1
Reproduce
Sustained load test with memory tracking across all 3 runtimes. Confirmed Node and JVM heaps stable — leak is in native.
PHASE 2
Profile
Enable MallocStackLogging, take heap snapshots, diff to find growing allocations with stack traces.
PHASE 3
Fix
Cache metrics as val fields — one allocation at init instead of per-request.
PHASE 4
Prove
Re-run same load test. Before/after heap comparison. Flat RSS confirmed.
Root Cause: Each request to JVM initiated a new set of OTel metrics instruments across JNI. On the Rust side, OTel instances are intentionally defined as long-lived thus with no cleaner hooked. While JVM GC freed the tiny wrapper, the native allocation was orphaned. At 5 req/s × 3 callbacks × ~10 metrics = hundreds of leaked allocations/sec.
Before: per-request allocation
// New Rust alloc every call
def eligibilityRequest(id) =
  metrics
    .counter("portals.request")
    .increment(tags)
After: cached singleton
// Created once at init
private val counter =
  metrics.counter("portals.request")

def eligibilityRequest(id) =
  counter.increment(tags)
03
Impact

Beyond the Fix

Unblocked Product Launch
The memory leak was a launch blocker for a new product feature built on nested callbacks. Identifying and fixing the issue unblocked the product team from shipping to production in time.
Framework Hardening
Identified a systemic gap in the framework: JNI wrappers for metrics lacked destructors. Filed and scoped long-term fix to add Drop/Cleaner to all native pointer wrappers, preventing future leaks for any capability author.
Cross-Functional Collaboration
Worked across the framework team and product team to coordinate debugging, validate the fix in staging, and establish native memory monitoring in production dashboards.
Knowledge Transfer
Ran a team walkthrough on cross-runtime memory analysis so others can independently debug similar issues.
Operational Guardrails
Established sustained load test patterns and add native RSS to the benchmark testing for leak detection.
Team Capability Building
Mentored engineers on Rust ownership semantics, native memory profiling with Instruments, and the principle that JVM GC does not manage native resources. Raised the team's debugging baseline for cross-runtime systems.
04

Cross-runtime bugs hide in the gaps between memory models.

The fix was a one-line change. Finding it required understanding three runtimes and their ownership boundaries. The real impact was turning one engineer's investigation into repeatable team knowledge.
